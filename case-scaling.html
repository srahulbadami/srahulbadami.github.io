Designing & Scaling a High Traffic Backend System

This system was responsible for handling high-volume traffic with strict latency requirements. During peak traffic, the system experienced p95 latency spikes, database contention, and intermittent failures. The goal was to redesign the architecture to support scalability, reliability, and fault tolerance.

📌 Requirements

- Handle thousands of requests per second - Maintain p95 latency < 200ms - Ensure high availability (99.9%+) - Support safe deployments

🏗️ High Level Design (HLD)

Client ↓ API Gateway ↓ Service Layer (Stateless) ↓ Cache (Redis-like) ↓ Database Async Flow: Service → Queue → Worker → DB
The system separates read-heavy and write-heavy workloads. API Gateway handles routing and throttling. Stateless services enable horizontal scaling. Caching reduces database pressure, and async queues handle heavy processing.

🔍 Low Level Design (LLD)

GET /orders/{id} → check cache → if miss → fetch from DB → update cache POST /orders → validate request → write to DB → publish event to queue → worker processes downstream tasks
Core logic focuses on minimizing synchronous dependencies and ensuring idempotency across operations.

🔌 API Design

GET /orders/{id}
Response: { id, status, amount }

POST /orders
Request: { userId, items, paymentInfo }
Response: { orderId, status }

POST /payments
Handles payment processing with idempotency key

⚠️ Failure Handling

Retry Mechanism:
Transient failures retried with exponential backoff.

Dead Letter Queue (DLQ):
Failed messages pushed to DLQ for manual inspection.

Circuit Breaker:
Stops calling failing services to prevent cascading failures.

Idempotency:
Ensures duplicate requests don’t create inconsistent state.

📉 Bottlenecks Identified

- Database contention - Blocking synchronous calls - Lack of caching

⚙️ Improvements

- Introduced caching layer - Moved heavy operations to async workers - Optimized DB queries and indexing - Added monitoring and alerts

📈 Impact

- Reduced p95 latency from ~220ms → ~150ms - Improved throughput significantly - Reduced system failures under peak load

🧠 Learnings

Scaling systems is about removing bottlenecks and designing for failure. Introducing asynchronous processing and caching provided the biggest improvements. Observability was critical in identifying system weaknesses.
